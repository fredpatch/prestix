## Sprint 8 Commission Divers — Correction Approval Workflow

- New table `commission_edit_requests` (+ `commission_edit_status` enum: pending/approved/rejected). Not in the original M10 spec — added per real business need (agent mistake correction without allowing silent direct edits, especially on `commissionAmount`).
- Considered three designs explicitly before building: (A) direct edit fully audited, (B) lock amount only, (C) full approval queue. Chose C per explicit decision.
- Service: `createEditRequest` (agent+, mandatory reason, blocks a second pending request on the same transaction), `approveEditRequest` (admin+, applies exact proposed patch atomically), `rejectEditRequest` (admin+, transaction untouched).
- List endpoint enriches each commission with `pendingEditRequestId` via one bulk query, not N+1.
- Frontend: `RequestCommissionEditDialog` (scoped to date/amount/note only — client/référent and type-specific fields deliberately out of this first pass), `CommissionEditQueuePage` (admin+, before/after diff table, requester name via users lookup, approve/reject with optional rejection note), pending badge on the commission list.

## Sprint 8 Commission Divers — Bugs Found

- Client `CommissionType` type never exposed `fieldSchema` despite the server already returning it — would have made the dynamic-field renderer silently render nothing. Caught before shipping the entry dialog.
- Used `<DialogTrigger asChild>` out of habit (Radix convention) — this project uses Base UI, which doesn't support that prop. Confirmed via grep that every other dialog in the codebase uses the plain `<DialogTrigger><Button/></DialogTrigger>` pattern; fixed to match.
